# AI 图片生成与编辑工作台开发计划

## 1. 开发策略

本项目按 `M1 -> M2 -> M3 -> M4 -> M5` 里程碑推进，优先完成可运行、可演示、可测试的 P0 生成闭环。每个开发任务完成后，必须更新 `docs/development-progress.md`，记录完成内容、验证方式、结果和遗留问题。

默认技术路线：

| 层级 | 技术选择 | 说明 |
| --- | --- | --- |
| 前端 | Vite + React + TypeScript | 适合快速构建工作台界面、组件化状态和响应式布局 |
| 后端 | Node.js + Express + TypeScript | 作为 BFF/API，承接配置、生成代理、错误归一化和后续存储能力 |
| 测试 | Vitest + Testing Library + Playwright 预留 | 单元测试先覆盖领域逻辑、适配器、校验、错误映射 |
| 存储 | localStorage/IndexedDB 起步，后端数据库预留 | P0 支持未登录基础生成和本地历史 |

## 2. 里程碑计划

| 里程碑 | 范围 | 目标 | 完成标准 |
| --- | --- | --- | --- |
| M1 P0 生成闭环 | 模型配置、基础路由、生成页、设置、上传、生成请求、结果、错误、基础历史 | 用户可未登录完成一次真实代理生成闭环 | DEV-P0-001 到 DEV-P0-012 通过基础验证 |
| M2 P0/P1 效率能力 | 批量抽卡、cURL 完整联动、模板库、zip 下载、埋点 | 提升批量创作和开发对接效率 | 批量抽卡可运行，cURL 与模型能力严格匹配 |
| M3 P1 资产复用 | 历史详情、素材模板、模型对比、图片识别 | 支持资产沉淀、复用和模型效果比较 | 历史/模板/对比/识图页面可用并有真实失败态或请求草稿验证 |
| M4 P2 灰度能力 | 高级设置、推理测试、R2/OSS、本地归档、运营入口 | 补齐高级用户与运营能力 | 灰度开关可控，不影响 P0 主流程 |
| M5 参考项目像素级迁移 | `GPT-Image2-Studio` 原生运行时、页面、静态素材、服务端接口和工作流 | 在当前项目中保留参考项目完整 UI/动画/功能入口 | 当前项目启动后可从菜单进入参考 Studio，独立运行时可完成原项目多模式创作、配置、画廊和记录管理 |

## 5. 进度记录规则

1. 每完成一个任务，更新 `docs/development-progress.md`。
2. 记录必须包含任务 ID、状态、完成内容、验证命令、验证结果、下一步。
3. 若任务部分完成，状态标记为 `进行中`，并写清楚剩余项。
4. 若测试无法运行，必须记录原因和替代验证方式。
5. 不把完整 API Key、真实用户图片、敏感提示词写入进度或日志。
