## 2. 产品概述

本产品是一个多模型 AI 图片生成、图片编辑和图像理解工作台。用户可以选择不同图像模型，输入提示词，上传参考图，设置尺寸、分辨率、数量和质量参数，生成图片并查看结果；同时支持批量生成、模型对比、历史记录、素材模板复用、图片识别和推理测试等辅助工具。

产品定位为面向设计师、内容创作者、电商运营、开发者和企业客户的 AI 图片 API 测试与创作平台。前台提供低门槛创作体验，后台通过 API Key、请求代码示例和日志/充值入口服务开发者集成场景。

## 3. 目标与非目标

### 3.1 产品目标

1. 提供一站式 AI 图片生成入口，覆盖文生图、图生图/参考图编辑、多图融合等场景。
2. 通过模型切换和参数化控制，让用户快速比较不同模型的速度、价格、画质和尺寸能力。
3. 通过提示词模板、素材模板和提示词优化降低创作门槛，提高复用效率。
4. 通过 cURL 请求代码、API Key 设置和端点文档支持开发者快速接入。
5. 通过批量生成、历史记录和打包下载支持高频生产任务。

### 3.2 非目标

1. 不在本期实现完整的图片资产管理系统，如团队空间、权限分组、复杂标签检索。
2. 不在本期实现支付、发票、账单中心，只保留充值/日志等外链入口。
3. 不在本期承诺所有模型都支持同一套参数，参数能力必须按模型配置动态展示。
4. 不在本期实现人工审核系统，但需要保留内容合规提示与失败状态处理。

#### 6.4.1 模型能力补充规则

1. 模型列表至少支持 16 个线上可见模型配置：Nano Banana Pro、Nano Banana 2、Nano Banana Lite、GPT Image 2 All、GPT Image 2 VIP、GPT Image 2、Nano Banana、SeeDream 5.0、SeeDream 4.5、Flux Kontext Pro、Flux Kontext Max、Flux 2 Pro、Flux 2 Max、Flux 2 Flex、Flux 2 Klein 9B、GPT Image 1.5。
2. 模型选择器需展示模型名称、速度标签、价格说明、新模型/热门标签和核心能力摘要；切换模型后同步更新上传上限、尺寸、分辨率、质量、输出格式、价格和 cURL。
3. 默认模型、已下线模型回退、模型临时限制和高分辨率禁用规则必须配置化；不得依赖前端硬编码判断。
4. 对只支持 1 张输出的模型，数量选择应固定为 1 或禁用多张；对支持多图融合的模型，应展示参考图上限，例如 Flux 2 系列最多 4 或 8 张。
5. 对 GPT Image 2 All 等无原生 size 参数的模型，尺寸比例可通过提示词标记引导，费用区需说明最终分辨率为估算或实测值。

### 6.5 尺寸限制提示

截图中 gpt-image-2-vip 存在尺寸限制公告：因上游调整，size 参数暂时失效，仅支持自适应 1K 出图，恢复后再开放多比例与 2K/4K。

要求：

1. 当模型存在临时能力限制时，在顶部公告条和参数区内同时提示。
2. 被限制的参数应置灰或固定为可用值，避免用户误以为可选择。
3. cURL 请求代码必须同步移除或固定受限参数。
4. 当限制解除后，通过配置恢复选项，不需要发版改代码。

### 6.7 生成流程

1. 用户选择模型。
2. 用户输入提示词，可选上传参考图。
3. 用户设置尺寸、分辨率、数量、质量等参数。
4. 系统校验必填项、文件格式、文件大小、API Key、模型能力和余额。
5. 用户点击开始生成。
6. 系统进入生成中状态，禁用重复提交。
7. 生成成功后在右侧结果区展示图片。
8. 生成失败时展示错误原因、重试入口和可排查建议。
9. 生成结果写入历史记录。

### 6.8 结果区

| 状态 | 展示要求 |
| --- | --- |
| 空态 | 展示占位图标和“生成的图片将在这里显示”类提示 |
| 生成中 | 展示进度、当前调用模型、预计等待时间或排队状态 |
| 成功 | 展示生成图片、下载、复制图片链接、查看原图、再次生成等操作 |
| 部分成功 | 多张生成时需区分成功和失败项 |
| 失败 | 展示错误摘要、错误码、重试按钮；不暴露敏感密钥 |

## 7. 对接代码需求

### 7.1 代码展示

生成页底部提供“对接代码 (cURL)”折叠面板，默认可展开查看当前配置对应的请求代码。

要求：

1. 请求代码随模型、提示词、参考图、尺寸、分辨率、数量、质量参数实时更新。
2. 默认使用占位密钥，例如 `sk-YOUR_API_KEY`。
3. 勾选“显示真实密钥”后才展示本地保存的 API Key。
4. 支持一键复制代码。
5. 支持跳转端点文档。
6. 不同模型使用不同端点和请求体结构。

### 7.2 端点适配

| 模型类型 | 示例端点 | 要求 |
| --- | --- | --- |
| GPT Image 系列 | `POST https://api.apiyi.com/v1/images/generations` | 请求体包含模型、提示词、尺寸、质量、数量等兼容字段 |
| Gemini Image 系列 | `POST https://api.apiyi.com/v1beta/models/{model}:generateContent` | 请求体包含 contents、parts、generationConfig 和 imageConfig |
| 其它模型 | 按模型配置 | 通过模型元数据驱动，不硬编码页面逻辑 |

适配补充：

1. GPT Image 2 All 默认走 Images API 并返回 `b64_json`，可配置回退到 chat/completions，但需提示外链破图风险。
2. OpenAI 官转类 `gpt-image-*` 端点不接受 `response_format` 时，代码生成器必须自动移除该字段，避免 400 Unknown parameter。
3. Gemini 原生端点需按 `contents.parts` 组织文本与参考图；参考图存在时需要根据模型能力选择 base64、URL 或文件上传策略。
4. 代码复制时默认使用占位 Key；勾选显示真实密钥仅影响本次展示，不应改变持久化安全策略。

## 16. 校验与异常

| 场景 | 处理要求 |
| --- | --- |
| 未输入提示词且未上传参考图 | 阻止提交，提示输入提示词或上传参考图 |
| API Key 缺失 | 引导到设置弹窗输入 API Key |
| 文件格式不支持 | 阻止上传并提示支持格式 |
| 文件超过大小限制 | 阻止上传并提示单图 20 MB 限制 |
| 参考图数量超限 | 阻止继续添加并提示上限 |
| 模型参数不兼容 | 自动禁用不可用参数并展示说明 |
| 余额不足 | 阻止提交并引导充值 |
| 上游失败 | 展示错误原因、错误码和重试入口 |
| 部分结果失败 | 成功图保留，失败任务可单独重试 |
| 内容违规 | 提示合规风险，不展示违规细节扩散 |
| Google 返回 candidatesTokenCount 为 0 | 展示“谷歌拒绝出图”，提示修改提示词或参考图 |
| API 返回 finishReason | 将 PROHIBITED_CONTENT、SAFETY、NO_IMAGE、RECITATION、MAX_TOKENS 等映射为用户可理解原因 |
| gpt-image-2 返回 429 | 提示可能需要切换到 `image2Enterprise` 企业分组，并给出说明入口 |
| 网络中断或 Failed to fetch | 提示本次可能未扣费，可重试，并建议切换网络或减少参考图 |
| 图片临时链接失效 | 保留历史元数据，提示重新生成、使用本地归档或及时保存 |
| 自定义存储连接失败 | 回退默认存储，并提示当前使用的存储类型 |

## 18. 权限与安全

1. API Key 必须默认脱敏展示，复制代码时默认使用占位密钥。
2. 显示真实密钥需要用户主动勾选。
3. 上传图片和提示词可能包含隐私信息，提交前应明确发送到 API 服务。
4. 用户生成内容需遵守法律法规，不得生成违法违规内容。
5. 生成失败和日志展示不得泄露完整 API Key。
6. 跨设备同步需依赖登录态，未登录仅保存在本设备。
7. API Key 存在浏览器本地时不得宣称强加密；若仅 Base64 或等价弱编码，需要在安全设计中标注风险。
8. 主生成 Key、图像理解 Key、自定义配置 Key 需要分开管理，避免误用和误展示。
9. 本地目录授权仅在安全上下文和支持的 Chromium 浏览器可用，权限被撤销时需自动降级。
10. 自定义 R2/OSS 配置可能包含访问凭据，需脱敏展示并提供清除入口。
11. 素材模板中的参考图若进入云端存储，应明确对象删除策略，避免删除元数据后遗留云端文件。

## 20. 待确认问题

1. 本项目是否需要完整复刻 API易的所有模型，还是先接入自有模型列表？
2. 图片生成结果是否需要落库，还是仅本地历史记录即可？
3. API Key 是用户自填，还是由系统后端统一代理和计费？
4. 是否需要登录体系；若需要，登录方式是否只支持 Google？
5. 图片文件是否上传到本项目服务端中转，还是前端直传第三方 API？
6. 生成图片的下载、分享、二次编辑、再次生成是否进入首期范围？
7. 合规审核由模型上游承担，还是本项目需要增加前置审核？
8. 是否需要首期支持本地文件夹自动归档；若支持，是否限定 Chrome/Edge？
9. 自定义 R2/OSS 是否作为正式功能交付，还是仅作为高级设置灰度开放？
10. 模板库是否完全复刻 96 个线上模板，还是只保留高频模板作为 MVP？
11. gpt-image-2 429 的 `image2Enterprise` 企业分组提示是否进入首期错误文案？
12. 图片 URL 的默认有效期、落库保存时长和删除策略需要产品、后端、安全共同确认。

## 21. 本轮检查后的补充建议

当前 PRD 已覆盖主流程和页面功能，但仍建议在评审前补齐以下决策型信息：

1. 成功指标：定义生成成功率、首图等待时间、P95 请求耗时、模板使用率、cURL 复制率、批量下载完成率等核心指标。
2. MVP 边界：明确 P0 是否只做生成页和批量抽卡，还是同步交付历史、素材模板、图片识别。
3. 后端边界：确认 API Key 由前端直连第三方、后端代理，还是混合模式；这会影响安全、计费、日志和合规责任。
4. 数据留存：确认生成结果、参考图、提示词、历史记录、素材模板在本地、R2/OSS、Supabase 中的保存时长和删除路径。
5. 配置后台：模型列表、公告条、临时限制、价格、模板库是否需要运营后台；若无后台，需定义配置文件发布流程。
6. 合规策略：明确内容审核由上游模型完全承担，还是本产品增加前置提示词/图片审核与申诉机制。
7. 质量验收：把“可用”改成量化标准，例如错误码覆盖率、模型配置更新不发版、下载成功率、模板加载兜底成功率。
